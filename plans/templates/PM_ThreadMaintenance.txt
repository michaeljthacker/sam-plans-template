ROLE: PM
TASK: ThreadMaintenance

Goal: Keep plans/thread.md short and actionable by pruning resolved content and promoting durable knowledge.

plans/thread.md is an append-only log — all other actions only append to it.
ThreadMaintenance is the SOLE action that may delete, restructure, or compress thread content.

Inputs:
- plans/thread.md
- plans/config.json (workflow configuration — check before pruning)
- plans/MILESTONE.md
- plans/DECISIONS.md (project-specific architectural choices)
- plans/STANDARDS.md (team-level rules portable across projects)
- plans/STATUS.md
- plans/state.json

Instructions:
1) Read plans/config.json to understand which steps are deferred or skipped
   (if config.json is missing or a key is absent, use its default):
   - If `code_review = "every_milestone"`: do NOT prune implementation details,
     review requests, or Q&A content from earlier phases in this milestone —
     Principal.CodeReview hasn't run yet and will need that context.
   - If `formal_approval = "every_milestone"`: do NOT prune phase summaries
     or acceptance-criteria notes — Human.PhaseApproval hasn't run yet.
   - If `documentation_update = "every_milestone"`: do NOT prune implementation
     details that Writer.DocumentationUpdate will need.
   - If any setting is `"every_phase"` or `"never"`, prune normally for those concerns.
2) Identify resolved questions/discussions in plans/thread.md.
3) Promote information:
   - Durable "going forward" items → DECISIONS.md (if project-specific) or STANDARDS.md (if the rule would carry to any project)
   - Milestone-specific implementation detail → MILESTONE.md
4) Compress resolved thread content into 1–3 bullet summaries.
5) Remove verbose back-and-forth from thread.md.
   After pruning, thread.md should contain: a brief resolved summary at the top,
   followed by any still-active/unresolved content.
6) Update plans/STATUS.md (mandatory) noting maintenance occurred.
7) Update plans/state.json:
   - last_action.action_id = PM.ThreadMaintenance
   - last_action.result = "ok"
   - Check context.notes for an "After ThreadMaintenance: proceed to ..." directive.
     If present, set next_action_id accordingly and remove the note from context.notes.
     If absent, set next_action_id appropriately based on lifecycle position.
   - Set pause_type based on who acts next:
     - If next_action_id starts with `Human.*`: pause_type = "decision"
     - Otherwise: pause_type = "continue"

Output:
- Brief summary
- Exact edits/patches
