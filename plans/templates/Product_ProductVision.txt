ROLE: Product
TASK: ProductVision

Goal: Transform the human's project idea into a concrete product vision — a root README.md and plans/BUILD.md that define what is being built, why, and at what scope.

This is the FIRST action in a new Build. The human has provided a project idea (in plans/thread.md or verbally). Your job is to draft the foundational artifacts.

Inputs:
- plans/state.json (routing + context)
- plans/config.json (workflow configuration — read `status_updates` for STATUS init)
- plans/thread.md (project idea, answers to setup questions)

Instructions:
1) Read plans/thread.md for the human's project idea and any setup context.
   If the thread does not yet contain sufficient project context, check for:
   - A one-sentence project goal
   - A Build B1 short name or description
   - A Milestone M1 goal
   If any of these are missing and cannot be inferred from the provided idea,
   set last_action.result = "blocked", pause_type = "decision",
   next_action_id = Human.ResolveBlocker,
   and add a blocker requesting the missing information.
2) Determine build `size` (recorded in BUILD.md frontmatter — see plans/FORMATS.md):
   - First, scan plans/thread.md for an explicit hint. Accept either an explicit
     `size: <value>` line OR natural-language signals such as "patch", "small fix",
     "dependency bump", "small feature", "full prototype", "MVP", "major build".
     If the human stated a size, respect it.
   - If no hint, infer from the project scope using these definitions:
     - `full` — multi-milestone work (2+ milestones, typical 3+). Full prototype or major feature.
     - `single-milestone` — substantial feature, single milestone, multiple phases.
     - `phase-only` — small feature; exactly one phase, multiple steps.
     - `step-only` — patch / dependency bump / small fix; one phase, 1–3 steps.
   - If genuinely ambiguous, prefer the smaller size; the human can grow it at
     Human.ApproveBuild or via Principal.PlanDiversion.
3) Generate the root README.md:
   - Project name and one-line description
   - What the project does (2–4 sentences)
   - Key features / capabilities (bullet list)
   - Tech stack (if determinable from the idea)
   - Getting started placeholder
   - Keep it concise — this is a living document that will be refined
4) Generate plans/BUILD.md:
   - Frontmatter at the top: `---\nsize: <chosen size>\n---`
   - Fill in the Build ID (from state.json, typically B1)
   - Purpose: what is being built and why
   - Scope: in-scope and out-of-scope items
   - Success criteria: measurable outcomes
   - Milestones — count must match the declared `size`:
     - `full`: propose M1, M2, … (typically 3+)
     - `single-milestone` / `phase-only` / `step-only`: propose exactly one milestone (M1)
   - Risks / assumptions
5) Initialize plans/STATUS.md (this action always creates the file — it is the
   one unconditional STATUS write in the lifecycle):
   - Read `status_updates` from plans/config.json (default: "pm_only" if missing).
   - If `status_updates = "never"`: write the disabled stub and stop:
     ```
     # STATUS

     _(STATUS updates disabled via config: status_updates=never)_
     ```
   - Otherwise, write the normal initial snapshot:
     - Top: `**Update configuration:** ` + the current `status_updates` value
     - Now section: Build ID, Milestone M1, no Phase yet
     - Blockers: None
     - Recent: "Product.ProductVision completed — README.md and BUILD.md drafted"
     - Next: "Principal.BuildReview to validate feasibility"
6) Update plans/state.json:
   - last_action.action_id = Product.ProductVision
   - last_action.result = "ok"
   - next_action_id = Principal.BuildReview
   - pause_type = "continue"
   - Update context.goal with a one-sentence summary of the project

Constraints:
- Do not implement any code.
- Do not create plans/MILESTONE.md (that is Principal.MilestonePlan's job).
- Keep the README and BUILD concise — they will be reviewed and refined.
- BUILD.md frontmatter MUST include a valid `size` value (`full`, `single-milestone`, `phase-only`, or `step-only`).
- The milestone count in BUILD.md MUST match the declared size (one milestone for everything except `full`).
- If the project idea is too vague to produce meaningful artifacts, set last_action.result = "blocked", pause_type = "decision", next_action_id = Human.ResolveBlocker, and add a blocker asking for clarification.

Output:
- Brief summary of what was generated
- Exact content for README.md, plans/BUILD.md, plans/STATUS.md, plans/state.json
