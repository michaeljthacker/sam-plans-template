ROLE: Principal
TASK: MilestonePlan

Goal: Draft plans/MILESTONE.md for the current milestone — a detailed, phased implementation plan with acceptance criteria that Staff can execute one Phase at a time.

Inputs:
- plans/state.json (routing + context — identifies which milestone to plan)
- plans/config.json (workflow configuration — read `status_updates`)
- plans/BUILD.md (overall Build scope and milestone list)
- plans/DECISIONS.md (project-specific architectural choices)
- plans/STANDARDS.md (team-level rules portable across projects)
- plans/STATUS.md (current snapshot)

Instructions:
1) Identify the current milestone from plans/state.json (milestone_id).
2) Read plans/BUILD.md for the milestone's high-level goal and scope, AND the
   `size` field from BUILD.md frontmatter. If frontmatter is missing entirely
   (legacy build), treat as `size: full` per plans/FORMATS.md. The size caps the
   phase/step count you may plan (see plans/FORMATS.md for the full table):
   - `full` / `single-milestone` — no phase/step cap (current behavior)
   - `phase-only` — exactly 1 phase; multiple steps allowed
   - `step-only` — exactly 1 phase; 1–3 steps total
   If the scope you've inferred from BUILD.md cannot fit the cap, do NOT silently
   exceed it. Instead, set last_action.result = "blocked", pause_type = "decision",
   next_action_id = Human.ResolveBlocker, and add a blocker asking the human to
   either grow `size` (via Principal.PlanDiversion or by re-running ProductVision)
   or trim scope.
3) Draft plans/MILESTONE.md. Match prose depth to `size` per plans/FORMATS.md
   § "Prose depth scales with size" — for `phase-only`, keep phase descriptions
   and acceptance criteria terse (a few bullets, not paragraphs); for `full` /
   `single-milestone`, use current depth.
   - Milestone ID and goal (one sentence)
   - Break the milestone into Phases (P1, P2, P3, …), respecting the size cap:
     - Each Phase has a clear "What" section describing the work
     - Each Phase has "Acceptance" criteria as a checklist (testable/verifiable)
     - Optionally break Phases into Steps (S1, S2, …) if the Phase is complex
   - Add Notes / Dependencies section if relevant
   - Follow the ID conventions: B1-M1-P1, B1-M1-P1-S1, etc.
4) Keep Phases small enough to implement in a single Staff.ImplementationExecution cycle.
   - Rule of thumb: each Phase should be completable in one focused session
   - If a Phase seems too large, split it into smaller Phases — unless the
     `size` cap forbids more phases, in which case raise it as a blocker per
     step 2 above
4a) **Engineering judgment.** As you draft the plan, apply these checks to each
    non-trivial capability the milestone introduces (auth, payments, email,
    analytics, search, file storage, observability, real-time, background jobs,
    etc.):
    - **Buy vs. build:** if a managed service or established OSS option exists
      and you're planning to build it instead, note *why* in the Phase's "What"
      section as a one-line breadcrumb (e.g., "Building a minimal email
      dispatcher here; Postmark/Resend deferred — confirm OK").
    - **One-way doors:** flag Phases that commit to choices hard to reverse
      (data model, vendor, license, platform).
    - **Industry-standard divergence:** if a Phase departs from the common
      default, say so explicitly so the human can challenge it.
    Pick the recommendation you'd actually pick. The goal is breadcrumbs, not
    an options menu — but the human must be able to spot the controversial
    choices without re-deriving them. Stay quiet on trivial or obvious choices.
5) Update plans/STATUS.md (config-gated):
   - Read `status_updates` from plans/config.json (default: "pm_only" if missing).
   - If `status_updates = "every_action"`:
     - Refresh the `**Update configuration:** status_updates=<value>` line at top
     - Add a bullet to Recent: "Principal.MilestonePlan completed — MILESTONE.md drafted for <Milestone ID>"
     - Update Next: "Human.ApproveMilestone to review scope"
   - Otherwise: SKIP the STATUS write. The PM.StatusUpdate that runs after
     Human.ApproveMilestone will refresh STATUS for the new milestone.
6) Update plans/state.json:
   - last_action.action_id = Principal.MilestonePlan
   - last_action.result = "ok"
   - next_action_id = Human.ApproveMilestone
   - pause_type = "decision"
   - phase_id = null (not yet started)

Constraints:
- Do not implement code.
- Do not modify plans/BUILD.md.
- Phases must be concrete and actionable — avoid vague descriptions.
- Acceptance criteria must be verifiable (not "looks good" but "unit tests pass for X").

Output:
- Brief summary of the milestone plan
- Exact content for plans/MILESTONE.md, plans/STATUS.md, plans/state.json
